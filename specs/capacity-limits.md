# Capacity Limits

## Status: Approved

## Problem

Agency has no awareness of host resource limits. Operators can start agents until the host runs out of memory, at which point Docker's OOM killer starts terminating containers unpredictably. Meeseeks exacerbate this — a parent agent spawning 5 meeseeks adds ~3.2 GB of memory pressure with no pre-flight check.

Docker's default subnet pool (~15 bridge networks with `/16` allocation) is a separate but related constraint. With per-agent networks, the pool exhausts around 10 concurrent agents.

## Design: Setup-Time Profiling + Count-Based Enforcement

### Capacity Profiling at Setup

`agency setup` profiles the host and writes a capacity config to `~/.agency/capacity.yaml`:

```yaml
# Generated by agency setup — edit to override
host_memory_mb: 16384
host_cpu_cores: 8
system_reserve_mb: 3276
infra_overhead_mb: 4300
max_agents: 13
max_concurrent_meeseeks: 13
agent_slot_mb: 640
meeseeks_slot_mb: 640
network_pool_configured: true
```

The computation:

1. Read host total memory and CPU cores
2. System reserve = 20% of total memory (OS, buffers, headroom) — minimum 2 GB
3. Infra overhead = sum of shared container memory limits (egress 256 MB + comms 256 MB + knowledge 256 MB + intake 128 MB + web-fetch 256 MB + web 64 MB + gateway-proxy 16 MB + embeddings 3,072 MB = ~4,300 MB). If embeddings is disabled (`KNOWLEDGE_EMBED_PROVIDER != ollama`), subtract 3,072 MB.
4. Available for agents = total memory − system reserve − infra overhead
5. `max_agents` = floor(available / agent_slot_mb)
6. `max_concurrent_meeseeks` = floor(available / meeseeks_slot_mb) — shared pool with agents, not additive. The combined count of running agents + meeseeks cannot exceed the total slot count.

The operator sees this output during setup:

```
Host capacity profile:
  Memory: 16 GB total, 3.3 GB system reserve, 4.3 GB infrastructure
  CPU: 8 cores (2 reserved for system)
  Agent capacity: 13 concurrent (640 MB + 2.5 CPU each)
  Meeseeks: share the same pool (640 MB + 1.5 CPU each)

Written to ~/.agency/capacity.yaml — edit to adjust limits.
```

### Enforcement Points

Two enforcement points, same check: count running agents + meeseeks against the configured maximums.

**1. Agent start (`agency start <agent>`)**

The gateway reads `~/.agency/capacity.yaml` and counts running agent containers (workspace containers with `agency.managed=true`). If running agents + running meeseeks >= `max_agents`, the start is rejected:

```
Error: cannot start agent "alice" — host capacity reached
  Running: 13 agents, 0 meeseeks (max: 13)
  Each agent requires 640 MB + 2.5 CPU
  Run "agency infra capacity" for details
  Edit ~/.agency/capacity.yaml to adjust limits
```

Running agents are never stopped. The check only gates new starts.

If `capacity.yaml` is missing or unparseable, agent starts are rejected with: "Capacity config not found — run `agency setup` to profile host resources." This is fail-closed per ASK Tenet 4.

Rejected agent starts are logged to the gateway audit trail with the capacity state at time of rejection (ASK Tenet 2).

**2. Meeseeks spawn (`spawn_meeseeks` tool)**

The enforcer relays the spawn request to the gateway. The gateway performs the same capacity check. If no slots remain, the spawn fails and the body runtime receives a tool error:

```json
{"error": "cannot spawn meeseeks: host capacity reached (13/13 slots used)"}
```

The parent agent can handle this gracefully — it's a normal tool error response.

Rejected meeseeks spawns are audit-logged with the same detail as rejected agent starts (ASK Tenet 2).

### Docker Network Pool Configuration

`agency setup` checks Docker's `default-address-pools` configuration. If not configured (Docker defaults to `/16` subnets), setup offers to configure `/24` subnets from a `/12` pool:

```
Docker network pool: default (limited to ~15 networks)
Recommended: /24 subnets from 172.16.0.0/12 (~4,000 networks)
Configure now? This requires a Docker daemon restart. [y/N]
```

If accepted, writes to `/etc/docker/daemon.json`:

```json
{
  "default-address-pools": [
    {"base": "172.16.0.0/12", "size": 24}
  ]
}
```

If the file already exists with other config, merges the `default-address-pools` key without disturbing existing settings.

Sets `network_pool_configured: true` in `capacity.yaml` after successful configuration.

### API

`GET /api/v1/infra/capacity` returns the capacity model plus current utilization:

```json
{
  "max_agents": 13,
  "max_concurrent_meeseeks": 13,
  "running_agents": 3,
  "running_meeseeks": 2,
  "available_slots": 8,
  "agent_slot_mb": 640,
  "meeseeks_slot_mb": 640,
  "host_memory_mb": 16384,
  "host_cpu_cores": 8,
  "system_reserve_mb": 3276,
  "infra_overhead_mb": 4300,
  "network_pool_configured": true
}
```

### CLI

`agency infra capacity` reads the API endpoint and displays:

```
Host Capacity:
  Memory: 16 GB (3.3 GB reserved, 4.3 GB infra)
  CPU: 8 cores (2 reserved)

Agents: 3/13 running (8 slots available, 2 used by meeseeks)
Meeseeks: 2 active (shares agent pool)
Networks: configured (/24 subnets, ~4000 available)

Config: ~/.agency/capacity.yaml
```

### Doctor Check

`agency admin doctor` gains a `host_capacity` check:

- **pass**: running agents + meeseeks < 80% of max
- **warn**: running agents + meeseeks >= 80% of max ("approaching capacity")
- **fail**: `capacity.yaml` missing or unparseable

And a `network_pool` check:

- **pass**: `network_pool_configured: true`
- **warn**: default Docker pool detected ("limited to ~15 networks, run agency setup to configure")

### Operator Overrides

`capacity.yaml` is operator-editable. The computed values from `agency setup` are defaults. Operators can:

- Lower `max_agents` if the host runs other workloads
- Raise `max_agents` if they're willing to over-subscribe (memory limits are per-container, so Docker handles contention)
- Set `max_concurrent_meeseeks` independently of agents
- Re-run `agency setup` to re-profile after hardware changes

The gateway re-reads `capacity.yaml` on SIGHUP (same as other config hot-reload). When reloaded values differ from the previous config, the old and new limits are logged to the audit trail (ASK Tenet 10 — constraint changes are tracked).

### What Doesn't Change

- Container resource limits in `defaults.go` stay as-is. Capacity limits prevent over-subscription; they don't resize containers.
- `DefaultMeeseeksLimit` (5 per parent) remains as a per-parent ceiling. The capacity check is an additional system-wide gate.
- No automatic scaling, no container resizing, no dynamic memory sampling. The model is static config + simple counting.

## ASK Tenets Addressed

- **Tenet 1 (Constraints are external and inviolable):** Capacity enforcement runs in the gateway, outside the agent boundary. Agents cannot perceive, influence, or circumvent the check.
- **Tenet 2 (Every action leaves a trace):** Rejected starts and spawns are audit-logged with full capacity state.
- **Tenet 4 (Enforcement failure defaults to denial):** Missing or corrupt capacity config rejects all starts (fail-closed).
- **Tenet 8 (Operations are bounded):** Agent and meeseeks concurrency is now constrained by default, not unlimited.
- **Tenet 10 (Constraint history is immutable and complete):** Capacity config changes are logged with old and new values on reload.
- **Tenet 19 (Delegation cannot exceed delegator scope):** Meeseeks consume from the same capacity pool as their parent agents.

## Implementation Scope

1. **Capacity calculator** — `internal/orchestrate/capacity.go`. Reads host resources (`/proc/meminfo` on Linux, `sysctl` on macOS), current container count, `capacity.yaml`. Pure computation. Testable with mock inputs.

2. **Setup profiling** — `agency setup` calls the calculator, writes `capacity.yaml`, shows the operator the profile. Offers Docker pool configuration.

3. **Start-time enforcement** — pre-flight check in the agent startup path. Reads capacity config, counts containers, rejects if at limit.

4. **Meeseeks enforcement** — same check in the meeseeks spawn path via the gateway.

5. **API endpoint** — `GET /api/v1/infra/capacity` returns the model.

6. **CLI command** — `agency infra capacity` reads and displays the endpoint.

7. **Doctor checks** — `host_capacity` and `network_pool` checks added to `agency admin doctor`.
