Episodic memory gives agents the ability to recall specific past interactions: what happened, when, why, what the outcome was, and what was surprising or notable. It complements procedural memory (reusable how-to patterns) and the knowledge graph (organizational facts) by capturing the narrative of individual task executions.

The knowledge graph already stores typed entities — people, systems, decisions, incidents, lessons. Procedural memory captures generalizable approaches. Neither captures the specific story of what happened during a particular task. An agent that handled a production incident last Tuesday cannot recall "the database failover took 45 minutes because the replica was 2 hours behind." That is episodic knowledge — not a fact, not a procedure.

This matters for three reasons:

1. **Recurring incidents need context from previous occurrences.** When INC-4567 fires and the agent recalls that INC-4501 two days ago was caused by the same deploy, it can skip the diagnostic phase and go straight to the known root cause.
2. **Agents interacting with the same stakeholders benefit from remembering past conversations.** Knowing that a particular oncall engineer responds in under 3 minutes changes escalation strategy.
3. **Patterns only emerge when you can recall specific episodes and compare them.** A procedural pattern like "check the deploy log first" only forms after enough episodes reinforce it.


## Episode Schema

An episode is a knowledge graph entity of type `episode`. Each episode records a single task execution:

```yaml
entity_type: episode
attributes:
  agent: henrybot900
  mission_id: 8a3f2b1c-4d5e-6f7a-8b9c-0d1e2f3a4b5c
  mission_name: ticket-triage
  task_id: task-abc123
  timestamp: 2026-03-27T10:00:00Z
  duration_minutes: 12
  outcome: success              # success | partial | failed | escalated
  summary: |
    Triaged INC-4567: production API latency spike affecting checkout flow.
    Root cause was a misconfigured connection pool after yesterday's deploy.
    Assigned P1, tagged @oncall-infra and @oncall-backend. Escalated within
    3 minutes. Incident resolved by oncall in 45 minutes via config rollback.
  notable_events:               # what was surprising, unusual, or worth remembering
    - "Connection pool misconfiguration was introduced in deploy #2847 — same deploy that caused INC-4501 two days ago"
    - "Oncall response was fast (3 min to acknowledge) — new PagerDuty routing is working"
  entities_mentioned:           # links to existing knowledge graph entities
    - { type: incident, name: "INC-4567" }
    - { type: system, name: "checkout-api" }
    - { type: person, name: "alex-oncall" }
  operational_tone: problematic  # routine | notable | problematic
  tags: [production-incident, api-latency, config-error, deploy-related]
```

### Field Details

- **`agent`** — Name of the agent that executed the task. Immutable.
- **`mission_id`** — UUID of the active mission at execution time. Empty if the task was ad-hoc (DM or @mention without a mission).
- **`mission_name`** — Human-readable mission name. Denormalized for display.
- **`task_id`** — The task ID from the `task_complete` signal. Links the episode to budget tracking and audit.
- **`timestamp`** — ISO 8601. When the task started.
- **`duration_minutes`** — Wall-clock duration from task start to `task_complete` signal.
- **`outcome`** — One of `success`, `partial`, `failed`, `escalated`. Derived from the task's completion state and any error signals emitted during execution.
- **`summary`** — Natural language narrative of what happened. 2-5 sentences. Written by the generation prompt, not the agent.
- **`notable_events`** — List of strings capturing what was surprising, unusual, or worth remembering. These are the details that differentiate this episode from routine execution. May be empty for uneventful tasks.
- **`entities_mentioned`** — References to knowledge graph entities involved in the task. Each entry has `type` and `name`. Used for relationship edges and retrieval queries.
- **`operational_tone`** — How the task went operationally: `routine` (smooth, unremarkable execution), `notable` (something interesting happened but not problematic), `problematic` (stressful, degraded, or error-prone execution). This is an operational assessment, not emotional valence.
- **`tags`** — Freeform labels for filtering and retrieval. Generated by the summarization prompt. No controlled vocabulary — patterns emerge organically.


## Ontology Addition

The `episode` entity type is added to the knowledge graph ontology:

```yaml
entity_types:
  episode:
    description: A specific task interaction recording what happened, when, and what was notable
    attributes:
      - agent
      - mission_id
      - mission_name
      - task_id
      - timestamp
      - duration_minutes
      - outcome
      - summary
      - notable_events
      - entities_mentioned
      - operational_tone
      - tags
```

### Relationships

Episodes form three types of relationships:

- **`episode --occurred_during--> mission`** — Links an episode to its mission. Created when `mission_id` is present.
- **`episode --involves--> [entity]`** — One edge per entity in `entities_mentioned`. Links the episode to the people, systems, incidents, and other entities involved.
- **`episode --followed_by--> episode`** — Temporal chain linking consecutive episodes for the same agent and mission. Created when storing a new episode — the previous episode in the chain gets a `followed_by` edge to the new one. This enables "what happened before/after" traversal.

Relationship creation is handled during episode storage. The gateway resolves `entities_mentioned` against existing knowledge graph entities. Only entities that exist in the graph at storage time get `involves` relationship edges. Unresolved references are stored as attributes on the episode (the `entities_mentioned` list) but do not create relationship edges. This avoids the complexity of a reconciliation mechanism for dangling references. If the referenced entity is created later, an operator or agent can re-link it manually via `contribute_knowledge`, but automatic reconciliation is deferred.


## Episode Generation

Episodes are generated via a lightweight LLM call after each `task_complete` signal. The body runtime sends the conversation history to a one-shot summarization prompt that extracts the episode structure.

### Cost Model

Episode generation is a platform cost, not agent budget. The call uses the same routing infrastructure as procedural memory generation. Both episode and procedure can be extracted in a single LLM call (one call, two output sections) to avoid duplicate cost.

### Generation Trigger

The generation sequence:

1. Agent calls `complete_task(summary=...)` — the task is done
2. Body runtime emits `task_complete` signal to `agent-signals.jsonl`
3. Runtime invokes the post-task generation prompt with the conversation history
4. The LLM returns both the procedural extract (if procedural memory is enabled) and the episode extract
5. The runtime stores the episode in the knowledge graph via the enforcer-mediated knowledge endpoint

If the generation call fails (LLM error, timeout), the episode is not created. No retry — the task itself is already complete. A warning is logged to `agent-signals.jsonl` with signal type `episode_generation_failed`.

### Generation Prompt

The generation prompt is appended to the conversation history as a system-level instruction:

```
You are extracting an episodic memory record from the conversation above.
This is NOT a reusable procedure — it is a record of what specifically happened
during this task, for future recall.

Return a JSON object with these fields:

{
  "summary": "2-5 sentence narrative of what happened. Be specific: names,
              numbers, timelines, root causes. Write as if briefing someone
              who needs to understand this incident in 30 seconds.",
  "notable_events": [
    "List things that were surprising, unusual, or worth remembering.",
    "Include connections to past events, unexpected outcomes, timing anomalies.",
    "Omit routine steps — only capture what makes THIS task different.",
    "Empty list if the task was entirely routine."
  ],
  "entities_mentioned": [
    {"type": "person|system|incident|decision|process|metric|goal", "name": "..."}
  ],
  "outcome": "success|partial|failed|escalated",
  "operational_tone": "routine|notable|problematic",
  "tags": ["freeform", "labels", "for", "filtering"]
}

Rules:
- Be factual. Do not editorialize or add interpretation beyond what happened.
- The summary should stand alone — someone reading it should understand the
  situation without seeing the conversation.
- For notable_events, focus on what would be useful to recall in a future
  similar situation. "The connection pool was misconfigured by the same deploy
  that caused INC-4501" is useful. "I triaged the ticket" is not.
- For entities_mentioned, only include entities that played a meaningful role.
  Do not list every system or person casually mentioned.
- Tags should be specific enough to be useful in searches. Prefer
  "config-error" over "error", "deploy-related" over "deployment".
```

The `agent`, `mission_id`, `mission_name`, `task_id`, `timestamp`, and `duration_minutes` fields are populated by the runtime from task metadata — the LLM does not generate them.


## Episode vs. Procedure

Both episodes and procedures are generated from the same task completion event. They capture different aspects of the same work:

| | Procedural Memory | Episodic Memory |
|---|---|---|
| **Purpose** | How to do this kind of task | What happened during this specific task |
| **Generalizable** | Yes — patterns, approaches, tools | No — specific names, dates, outcomes |
| **Example** | "For P1 incidents, check deploy log first, then connection pool config" | "INC-4567 was caused by deploy #2847's connection pool misconfiguration, same deploy as INC-4501" |
| **Lifespan** | Evolves through reinforcement | Immutable after generation |
| **Retrieval** | By task type / mission | By time, entity, tag, semantic similarity |

Both are extracted in the same post-task LLM call. The generation prompt includes sections for both outputs. If only one memory type is enabled, only that section is requested.


## Episode Retrieval

Two retrieval paths surface episodes to agents.

### Session Start Injection

At session start, the runtime queries recent episodes for the current mission and injects them into the system prompt. This gives the agent immediate context about its recent work.

Query: episodes where `agent` matches the current agent AND `mission_id` matches the current mission, ordered by `timestamp` descending, limited to `max_retrieved` (default 5).

The injected block appears after procedural memory in the prompt assembly order:

```
identity.md
mission instructions
mission behavioral frame
mission knowledge (org facts)
procedural memory (how-to patterns)
episodic memory (recent episodes)        <-- NEW
memory index (agent private memory)
comms context
FRAMEWORK.md
AGENTS.md
skills
```

The injected format is compact to minimize token cost:

```
## Recent Episodes

### 2026-03-27 10:00 -- INC-4567 (P1, success, 12 min)
Production API latency spike affecting checkout flow. Root cause: misconfigured
connection pool from deploy #2847. Same deploy caused INC-4501 two days ago.
Notable: oncall response was fast (3 min) -- new PagerDuty routing working.

### 2026-03-26 14:30 -- INC-4550 (P3, success, 4 min)
Non-critical alert for elevated 404 rates on /api/v2/search. Confirmed
expected behavior after endpoint deprecation last Friday. Auto-resolved.

### 2026-03-26 09:15 -- INC-4548 (P2, escalated, 28 min)
Memory pressure on checkout-api pods. Horizontal scaling helped temporarily
but root cause was a memory leak in the session cache. Escalated to platform
team. Notable: same symptom pattern as INC-4400 two weeks ago -- session
cache is a recurring problem area.
```

If no episodes exist for the current mission (new mission, or first task), the section is omitted.

### XPIA Scanning

Episode summaries and notable events are LLM-generated content from past agent conversations. Like procedural memory, this content could contain adversarial input if a past task processed malicious data. All episode content MUST pass through XPIA scanning before injection into the system prompt. See the Procedural Memory spec for the full XPIA scanning protocol — the same mechanism applies to episodic memory injection.

### On-Demand Tool

The `recall_episodes` tool lets agents search their episodic memory during a task. This is useful when the agent encounters something that reminds it of a past event and wants to retrieve the full context.

```
recall_episodes(
  query: str,              # semantic search query (required)
  agent: str = None,       # filter by agent name (default: self)
  mission: str = None,     # filter by mission name
  from_date: str = None,   # ISO 8601 date, e.g. "2026-03-20"
  to_date: str = None,     # ISO 8601 date, e.g. "2026-03-27"
  entity: str = None,      # filter by entity name in entities_mentioned
  tag: str = None,         # filter by tag
  outcome: str = None,     # filter by outcome
  limit: int = 10          # max results
) -> list[Episode]
```

The tool queries the knowledge graph with the provided filters. The `query` parameter drives semantic similarity matching against the `summary` and `notable_events` fields. Structured filters (agent, mission, from_date, to_date, entity, tag, outcome) are applied as pre-filters before semantic ranking. Date parameters use ISO 8601 format — the agent converts natural language time references ("last week") to concrete dates in its reasoning before calling the tool.

Results are returned in relevance order (semantic similarity score), with ties broken by recency.

The tool respects authorization scope: an agent can query its own episodes and episodes from agents on the same mission. Cross-team queries require explicit authorization (ASK tenet 24).


## Episode Retention

Episodes accumulate over time. Retention policy balances recall depth with storage cost.

- **0-90 days** — All episodes retained in full. Queryable by all retrieval paths.
- **After 90 days** — Episodes are consolidated into monthly narrative summaries per mission. A single `narrative` entity in the knowledge graph covers one month of work for one agent on one mission.
- **Monthly narratives** — Kept indefinitely. They are organizational knowledge (ASK tenet 23), not transient agent state.

### Consolidation

Consolidation runs as a background gateway task on a daily schedule. For each agent+mission pair with episodes older than 90 days:

1. Query all episodes from the target month
2. Send to a summarization prompt that produces a monthly narrative
3. Store the narrative as a `narrative` entity in the knowledge graph
4. Delete the individual episode entities

The consolidation prompt emphasizes patterns and themes over individual incidents:

```
Summarize these {count} episodes from {agent} on mission {mission_name}
for {month} {year}. Focus on:
- Recurring themes and patterns
- Notable incidents that should be remembered long-term
- Changes in operational dynamics over the month
- Entities that appeared frequently

Write 1-2 paragraphs. This summary replaces the individual episodes, so
include anything worth remembering.
```

Monthly narratives link to the mission via `narrative --summarizes--> mission` and retain `agent` and time range as attributes.


## Configuration

Episodic memory is configured per-agent or per-mission in the agent config or mission YAML:

```yaml
episodic_memory:
  enabled: true              # default: true
  capture: true              # generate episodes on task_complete
  retrieve: true             # inject recent episodes at session start
  max_retrieved: 5           # episodes in system prompt
  retention_days: 90         # before consolidation
  tool_enabled: true         # expose recall_episodes tool
```

All fields are optional. Defaults apply when omitted.

- **`enabled`** — Master switch. When `false`, no episodes are generated or retrieved. Default `true`.
- **`capture`** — Generate episodes on task completion. Can be disabled for low-value repetitive tasks where episode history adds noise. Default `true`.
- **`retrieve`** — Inject recent episodes into the system prompt at session start. Can be disabled to reduce prompt size. Default `true`.
- **`max_retrieved`** — Maximum number of episodes injected at session start. Higher values give more context but consume more tokens. Default `5`.
- **`retention_days`** — Days before episodes are consolidated into monthly narratives. Default `90`.
- **`tool_enabled`** — Expose the `recall_episodes` tool to the agent. Default `true`.

Mission-level config overrides agent-level config. If a mission sets `episodic_memory.max_retrieved: 3`, that takes precedence over the agent's default of 5.


## Privacy and Authorization

Episodes are agent-scoped by default. Access rules:

- An agent can query its own episodes without restriction.
- Team members on the same mission can query each other's episodes. Shared mission context justifies shared episode access — if two agents are both triaging tickets, knowing what the other handled prevents duplicate work.
- Cross-team episode access requires explicit authorization via the existing knowledge graph authorization model.
- Episodes may reference sensitive incident details. The knowledge graph's entity-level authorization applies to episode entities the same way it applies to any other entity type.

Episodes are immutable after generation. The agent that generated an episode cannot edit or delete it. This ensures episodic memory is a reliable audit-adjacent record, not a malleable narrative the agent can revise (ASK tenet 23 — organizational knowledge is durable infrastructure, not agent state).

**Tenet 12 note (synthesis cannot exceed individual authorization):** An agent on a shared mission can query teammates' episodes. Episode summaries may reference systems or incidents that the querying agent cannot directly access. This is by design — episode content is organizational knowledge shared within mission scope, not a capability grant. The querying agent learns *about* a system from the episode narrative but cannot *access* that system unless it has the corresponding capability. This is the same information-sharing model as the knowledge graph itself.


## REST API

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/api/v1/agents/{name}/episodes` | List episodes for an agent. Supports query params: `mission`, `from`, `to`, `outcome`, `tag`, `limit`, `offset` |
| `GET` | `/api/v1/missions/{name}/episodes` | List episodes across all agents on a mission. Same query params |

Both endpoints return paginated results ordered by `timestamp` descending. Response format:

## CLI

Episodic memory extends existing commands rather than adding new top-level commands:

| Command | Change |
|---------|--------|
| `agency mission knowledge <name>` | Add episode count and date range to output |
| `agency show <agent>` | Add total episode count and most recent episode summary |

No dedicated `agency episodes` command. Episodes are organizational knowledge stored in the graph — they are queried through the REST API, the `recall_episodes` tool, or the existing `agency admin knowledge` commands.


## Task Tier Interaction

Same tier gates as procedural memory:

- **Capture** — activates at `standard` and `full` tiers. Episodes are generated in the same post-task LLM call as procedures (zero additional cost). Skipped at `minimal`.
- **Retrieval/injection** — activates at `full` tier only. Episode summaries add ~300-800 tokens to the system prompt.
- **recall_episodes tool** — available at `standard` and `full` tiers. Agents can explicitly search episodic memory even when injection is off.

When `cost_mode: frugal`, all episodic memory is disabled. When `cost_mode: balanced`, capture is on and retrieval is on. When `cost_mode: thorough`, capture + retrieval + tool all on with higher `max_retrieved`.

See the Task Tier and Cost Model spec for the full tier classification logic.


## ASK Compliance

- **Tenet 23 (Organizational knowledge is durable infrastructure)** — Episodes persist in the knowledge graph beyond agent lifecycle. Killing an agent does not delete its episodes. Monthly narrative consolidation preserves long-term knowledge.
- **Tenet 24 (Knowledge access bounded by authorization)** — Episode retrieval respects agent scope, team scope, and cross-team authorization. The `recall_episodes` tool enforces the same access rules as any knowledge graph query.
- **Tenet 3 (Mediation is complete)** — Episode storage and retrieval go through the enforcer-mediated knowledge endpoint. No direct graph access from the agent.
- **Tenet 2 (Every action leaves a trace)** — Episode generation and retrieval are logged in the enforcer audit trail. The `episode_generation_failed` signal provides visibility when generation fails.
- **Immutability** — Episodes cannot be edited or deleted by agents. Only the consolidation process (a gateway background task, outside the agent boundary) can replace episodes with narrative summaries.
