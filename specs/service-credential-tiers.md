# Scoped Service Credentials

**Status:** Partially implemented (Phase 1 in progress)
**Last updated:** 2026-04-01

**Implementation notes:** The credential store (`internal/credstore/`) and `agency creds` CLI are implemented. Enforcer scope checking (`CheckScope` in `images/enforcer/services.go`) is implemented — the enforcer validates tool-to-scope mappings at request time. The enforcer does scope checking only and never holds or injects real credentials; credential resolution is handled by the egress proxy via `SocketKeyResolver` (gateway Unix socket). Service manifests are generated by a unified manifest generator (`internal/api/manifest.go`). Phase 2 (scope profiles, multi-key) and Phase 3 (compensating controls, automatic profile computation) are not yet implemented.

## Problem

All agents sharing a service (e.g., `limacharlie-api`) get the same API key
with the same scopes. The detection-tuner only reads detection history but
gets a key that can create D&R rules, isolate sensors, and delete rules.
This violates ASK Tenet 4 (least privilege).

The operator has no way to know what scopes each agent actually needs. They
create one key with everything checked, grant it to all agents, and hope for
the best. When an external service requires separate keys with different
scopes (like LimaCharlie), there's no mechanism to express that.

## Design Goals

1. Agent presets declare the specific external scopes they need
2. Required vs optional scopes — agents degrade gracefully without optional ones
3. Team/pack-level base scopes reduce per-agent key sprawl
4. The platform tells the operator exactly what keys to create
5. Enforcer validates scopes at request time — not advisory, enforced

## ASK Tenet Alignment

| Tenet | Requirement | How this spec addresses it |
|-------|-------------|---------------------------|
| 1 (External constraints) | Agents cannot influence scope declarations | Scope declarations are operator-authored preset files outside the agent boundary. Agents have no read/write access. |
| 2 (Audit trail) | Scope checks are auditable | Enforcer logs every scope check (pass and fail), profile changes, tool filtering decisions, and grant operations. |
| 3 (Complete mediation) | No bypass path for scope restrictions | Enforcer validates tool→scope at request time. Tool filtering is defense-in-depth, not the enforcement boundary. |
| 4 (Least privilege) | Agents get only the scopes they need | Preset declarations + tool-to-scope mapping + enforcer validation. Profile merging is a documented tradeoff with doctor warnings. |
| 5 (No blind trust) | Trust relationships visible | Doctor shows full scope inheritance chain. Platform cannot verify external key scopes — flagged as operator responsibility. |
| 6 (Atomic constraints) | Profile changes are atomic | Enforcer SIGHUP reloads atomically. In-flight requests complete under old profile. New profile takes effect on next request boundary. |
| 11 (Delegation bounds) | Meeseeks cannot exceed parent scopes | Meeseeks inherit parent's scope profile. Spawn rejects if requested scopes exceed parent. |
| 15 (No self-elevation) | Agents cannot change their own scopes | No agent-accessible API can modify scope profiles or grants. Elevation requires operator action. |
| 17 (Verified principals) | External sources cannot inject scope declarations | Scope resolution uses only locally-stored, operator-approved artifacts. Hub installs flagged for review. |

## Current Architecture

```
preset.yaml (hub)
  requires.credentials:
    - name: limacharlie-api          ← opaque, no scope info
      grant_name: limacharlie-api

credential store (~/.agency/credentials/store.enc)
  LIMACHARLIE_API_KEY → encrypted    ← one key, manually scoped, set via `agency creds set`

registry/services/limacharlie-api.yaml
  credential.env_var: LIMACHARLIE_API_KEY

enforcer (per-agent)
  ← loads only granted services from agent's services.yaml
  ← validates scope via CheckScope on X-Agency-Service header
  ← passes header through to egress (enforcer never holds real keys)

egress proxy (shared)
  ← resolves credentials via SocketKeyResolver (gateway Unix socket)
  ← different key_ref = different key (already works)
```

## Solution: Scope Declarations in Agent Definitions

### 1. Preset-Level Scope Declarations

Agent presets declare what external scopes they need per service. These
declarations are **operator-authored artifacts outside the agent boundary**.
Agents have no read or write access to preset files, scope profiles, or
any scope resolution inputs. This is enforced by the existing container
filesystem isolation — preset files live on the host, not inside the
agent workspace.

```yaml
# presets/detection-tuner/preset.yaml
requires:
  credentials:
    - name: limacharlie-api
      grant_name: limacharlie-api
      scopes:
        required:
          - insight.det.get    # must have: read detections
          - sensor.get         # must have: sensor info for context
          - dr.list            # must have: see existing rules
          - fp.ctrl            # must have: list existing FP rules
          - insight.stat       # must have: detection statistics
        optional:
          - insight.evt.get    # nice to have: timeline deep-dives
```

```yaml
# presets/detection-engineer/preset.yaml
requires:
  credentials:
    - name: limacharlie-api
      grant_name: limacharlie-api
      scopes:
        required:
          - insight.det.get    # read detections
          - sensor.get         # sensor context
          - dr.list            # list existing rules
          - dr.set             # create/update rules
          - dr.del             # delete rules
          - fp.ctrl            # create/manage FP rules
        optional:
          - sensor.task        # send commands for testing
          - insight.evt.get    # timeline queries
          - ext.request        # IOC search via extensions
```

```yaml
# presets/fleet-manager/preset.yaml
requires:
  credentials:
    - name: limacharlie-api
      grant_name: limacharlie-api
      scopes:
        required:
          - sensor.get         # sensor info
          - sensor.list        # list all sensors
          - insight.det.get    # detection context
        optional:
          - sensor.tag         # tag sensors
          - org.conf.get       # org config
          - insight.list       # insight status
```

```yaml
# presets/threat-hunter/preset.yaml
requires:
  credentials:
    - name: limacharlie-api
      grant_name: limacharlie-api
      scopes:
        required:
          - insight.det.get    # read detections
          - insight.evt.get    # timeline/event history
          - sensor.get         # sensor info
        optional:
          - sensor.task        # live sensor queries
          - ext.request        # IOC search
          - insight.stat       # detection stats
```

### 2. Scope Profiles

When multiple agents need similar scopes, avoid creating N keys. Group agents
into **scope profiles** — a named set of scopes that maps to one API key.

```yaml
# ~/.agency/infrastructure/scope-profiles.yaml
profiles:
  limacharlie-api:
    # Profile name → env var for the key
    security-read:
      env_var: LC_API_KEY_READ
      scopes:
        - insight.det.get
        - insight.evt.get
        - insight.stat
        - insight.list
        - sensor.get
        - sensor.list
        - sensor.tag
        - dr.list
        - fp.ctrl            # LC bundles read+write in one scope
        - ext.request
        - org.conf.get
      agents:
        - detection-tuner
        - alert-triage
        - fleet-manager
        - threat-hunter

    security-write:
      env_var: LC_API_KEY_WRITE
      scopes:
        - insight.det.get
        - insight.evt.get
        - insight.stat
        - sensor.get
        - sensor.task
        - dr.list
        - dr.set
        - dr.del
        - fp.ctrl
        - ext.request
      agents:
        - detection-engineer
```

The platform computes profiles automatically from agent scope declarations:
agents with identical required scopes share a profile. The operator can
override with explicit profile assignments.

**Known tradeoff (Tenet 4):** Profile merging unions scopes across agents
in the same profile. Detection-tuner doesn't need `sensor.tag` but gets a
key that has it because fleet-manager is in the same profile. `agency admin
doctor` flags agents whose effective profile scopes exceed their declared
required+optional scopes. Operators who need strict per-agent isolation
should create narrower profiles.

### 3. Key Provisioning Flow

#### `agency doctor` — Scope audit

```
$ agency admin doctor

  limacharlie-api:
    Profile: security-read (LC_API_KEY_READ)
      Agents: detection-tuner, alert-triage, fleet-manager, threat-hunter
      Required scopes: insight.det.get, sensor.get, sensor.list, dr.list,
                       fp.ctrl, insight.stat, insight.list, org.conf.get
      Optional scopes: insight.evt.get, sensor.tag, ext.request
      ⚠ Key not provisioned — run: agency creds set --name LC_API_KEY_READ ...
      ⚠ detection-tuner: profile grants 6 scopes beyond declared need
         (sensor.list, sensor.tag, insight.list, org.conf.get, ext.request,
          insight.evt.get)

    Profile: security-write (LC_API_KEY_WRITE)
      Agents: detection-engineer
      Required scopes: insight.det.get, sensor.get, dr.list, dr.set,
                       dr.del, fp.ctrl
      Optional scopes: sensor.task, insight.evt.get, ext.request
      ⚠ Key not provisioned — run: agency creds set --name LC_API_KEY_WRITE ...

    Unscoped agents (all tools available — no scope declarations):
      ⚠ security-explorer — add scopes to preset for least-privilege

    Note: Platform cannot verify external key scopes. Confirm the key
    created in the LC console has the scopes listed above.
```

#### `agency grant` — Scope-aware grants

```
$ agency grant detection-tuner limacharlie-api

Resolved profile: security-read (LC_API_KEY_READ)
Required scopes for detection-tuner:
  ✓ insight.det.get
  ✓ sensor.get
  ✓ dr.list
  ✓ fp.ctrl
  ✓ insight.stat
Optional (verify in LC console):
  ? insight.evt.get

```

```
$ agency grant --dry-run detection-engineer limacharlie-api

Would assign profile: security-write (LC_API_KEY_WRITE)
Effective scopes (profile union):
  insight.det.get, insight.evt.get, insight.stat, sensor.get,
  sensor.task, dr.list, dr.set, dr.del, fp.ctrl, ext.request

  insight.evt.get (optional), ext.request (optional)

```

If no profile exists yet, the grant command creates one:

```
$ agency grant detection-tuner limacharlie-api

No scope profile covers detection-tuner's requirements.
Creating profile 'limacharlie-api-detection-tuner':
  Env var: LC_API_KEY_DETECTION_TUNER
  Required scopes: insight.det.get, sensor.get, dr.list, fp.ctrl, insight.stat

Create a key in LimaCharlie with these scopes and store it:
  agency creds set --name LC_API_KEY_DETECTION_TUNER --value <your-key> \
    --kind service --scope agent:detection-tuner
```

### 4. Pack/Team-Level Defaults

Packs can declare a base scope profile that all their agents inherit:

```yaml
# packs/security-ops/pack.yaml
name: security-ops
description: SOC agent team for LimaCharlie

requires:
  credentials:
    - name: limacharlie-api
      base_scopes:
        - insight.det.get
        - sensor.get
        - dr.list
```

Individual agent presets within the pack then only declare their _additional_
scopes:

```yaml
# Within the pack's detection-engineer preset
requires:
  credentials:
    - name: limacharlie-api
      scopes:
        required:
          - dr.set        # additional: create rules
          - dr.del        # additional: delete rules
          - fp.ctrl       # additional: manage FP rules
```

The platform merges: `pack base_scopes` + `preset required scopes` = full
required scope set for that agent.

Pack `base_scopes` are visible in `agency admin doctor` output, showing the
full inheritance chain: `pack base → preset required → preset optional →
effective profile`. Hub-installed packs have their scope declarations flagged
for operator review before activation (see Security Invariants below).

### 5. Tool-to-Scope Mapping

Service definitions declare which scope each tool needs:

```yaml
# registry/services/limacharlie-api.yaml
tools:
  - name: list_detections
    scope: insight.det.get
    method: GET
    path: /v1/insight/${LC_ORG_ID}/detections
    ...

  - name: create_fp_rule
    scope: fp.ctrl
    method: POST
    path: /v1/hive/fp/${LC_ORG_ID}/{name}/data
    ...

  - name: create_dr_rule
    scope: dr.set
    method: POST
    path: /v1/hive/dr-general/${LC_ORG_ID}/{name}/data
    ...

  - name: isolate_sensor
    scope: sensor.task
    method: POST
    path: /v1/${sensor_id}/task
    ...
```

This enables:

1. **Automatic tool filtering** (defense-in-depth) — when building the
   services-manifest for an agent, only include tools whose `scope` is
   covered by the agent's profile. Detection-tuner never sees
   `create_dr_rule` because it doesn't have `dr.set`. This is a visibility
   control, not the enforcement boundary.

2. **Enforcer scope validation** (enforcement boundary) — the enforcer
   checks tool→scope at request time. Even if an agent crafts a raw HTTP
   request, the enforcer blocks it if the tool's scope isn't in the agent's
   profile. See section 6.

3. **Compensating controls for bundled scopes** — when an external service
   bundles read+write into one scope (e.g., LC's `fp.ctrl`), the tool-to-
   scope mapping lets the enforcer restrict at the tool level. An agent
   with `fp.ctrl` in its profile but only `list_fp_rules` in its manifest
   can list FP rules but not create them, even though the underlying API
   key permits both.

### 6. Runtime Scope Validation (Enforcer)

The enforcer validates tool→scope at request time. This is the enforcement
boundary — not optional, not advisory.

```
Agent calls create_fp_rule → X-Agency-Service: limacharlie-api
Enforcer checks:
  1. Service granted? Yes (limacharlie-api in services.yaml)
  2. Tool identified from request path/method
  3. Tool scope: fp.ctrl
  4. Agent's allowed scopes: [insight.det.get, sensor.get, dr.list, fp.ctrl]
  5. fp.ctrl in allowed scopes? Yes → inject credential and forward
```

If the tool requires a scope not in the agent's allowed set, the enforcer
returns 403 with an actionable error:

```json
{
  "error": "scope_denied",
  "tool": "create_dr_rule",
  "required_scope": "dr.set",
  "agent": "detection-tuner",
  "message": "Agent does not have dr.set scope for limacharlie-api"
}
```

**Fallback for unscoped tools:** If a tool has no `scope` field, the
enforcer allows any granted service request (preserves backward
compatibility). `agency admin doctor` flags tools without scope annotations.

### Audit Requirements

All scope-related operations produce audit entries. Audit is written by
mediation (enforcer/gateway), never by agents (ASK Tenet 2).

| Event | Written by | Contains |
|-------|-----------|----------|
| Scope check pass | Enforcer | agent, service, tool, scope, timestamp |
| Scope check fail (403) | Enforcer | agent, service, tool, required_scope, allowed_scopes, timestamp |
| Profile created/modified | Gateway | profile_name, service, old_scopes, new_scopes, operator, timestamp |
| Profile agent assignment | Gateway | profile_name, agent, operator, timestamp |
| Grant applied | Gateway | agent, service, profile, scopes, operator, timestamp |
| Grant revoked | Gateway | agent, service, profile, operator, timestamp |
| Tool filtered from manifest | Gateway | agent, service, tool, required_scope, reason, timestamp |
| Hub component scope review | Gateway | component, service, declared_scopes, operator_action, timestamp |

## Security Invariants

These are non-negotiable design constraints, not implementation details.

1. **Agents cannot influence scope resolution.** Scope declarations live in
   preset files on the host filesystem, outside the agent container boundary.
   Agents have no read or write access to presets, scope profiles, service
   definitions, or credential store. The scope resolution chain uses only
   locally-stored, operator-approved artifacts.

2. **No agent-accessible API can modify scopes.** No REST endpoint, MCP
   tool, or body runtime function can create, modify, or delete scope
   profiles, grants, or tool-to-scope mappings. All scope changes require
   operator CLI or REST API calls authenticated with the gateway token.

3. **Scope elevation requires human approval.** Reducing scope can be
   automated (e.g., revoking a grant on policy violation). Increasing scope
   always requires explicit operator action (ASK Tenet 15).

4. **Meeseeks inherit parent scope ceiling.** A meeseeks spawned by an
   agent inherits at most the parent's scope profile for each service.
   The `spawn_meeseeks` handler validates: if the meeseeks preset declares
   scopes exceeding the parent's profile, the spawn is rejected with a
   clear error. Meeseeks cannot be granted additional scopes beyond what
   their parent holds (ASK Tenet 11).

5. **Profile changes are atomic.** Enforcer SIGHUP reloads the full scope
   configuration atomically — the enforcer operates under the old profile
   or the new one, never a mix. In-flight requests complete under the old
   profile. New profile takes effect on the next request boundary. The
   reload is logged with before/after state (ASK Tenet 6).

6. **Hub-installed components require scope review.** When `agency hub
   install` or `agency hub upgrade` introduces new or changed scope
   declarations, `agency admin doctor` flags them with a `NEW_SCOPES`
   warning. The operator must acknowledge new scope declarations before
   the component's agents can be granted the new scopes. This prevents
   a compromised hub component from silently expanding scope (Tenet 17).

7. **External API responses never feed into scope resolution.** Scope
   assignment uses only static, deploy-time artifacts. No runtime data
   from external APIs, connector poll results, knowledge graph queries,
   or agent tool outputs can influence which scopes an agent holds.

## Implementation Phases

### Phase 1: Declarations + Visibility + Enforcement

1. Add `scopes.required` / `scopes.optional` to preset credential blocks
2. Add `scope` field to service tool definitions
3. `agency admin doctor` computes and displays scope profiles, flags
   unscoped agents and tools, shows inheritance chains
4. `agency grant` shows required scopes and tells operator what key to create
5. `agency grant --dry-run` shows effective scope set without applying
6. Automatic tool filtering in services-manifest generation (defense-in-depth)
7. Enforcer scope validation at request time (enforcement boundary)
8. Audit logging for all scope events

### Phase 2: Scope Profiles + Multi-Key

1. `scope-profiles.yaml` config file
2. Multiple env vars per service (`LC_API_KEY_READ`, `LC_API_KEY_WRITE`)
3. `agency grant` auto-assigns agent to matching profile
4. Profile-aware credential resolution in enforcer
5. Pack-level `base_scopes` inheritance
6. Hub component scope review gate

### Phase 3: Advanced

1. Compensating tool-level controls for bundled external scopes
2. Automatic profile computation from agent declarations
3. Scope drift detection (agent's effective scopes vs declared needs)

## What Already Works

- Encrypted credential store (`~/.agency/credentials/store.enc`) with `agency creds` CLI (set, list, show, delete, rotate, test, group create)
- Credential groups for organizing credentials by scope (e.g., `agent:detection-tuner`)
- Egress credential resolution via `SocketKeyResolver` (gateway Unix socket) — no flat env files
- Enforcer scope checking (`CheckScope`) — validates tool-to-scope at request time
- Enforcer does scope checking only, never holds or injects real keys
- Per-agent service grants (constraints.yaml + services.yaml)
- Per-agent tool filtering via unified manifest generator (`internal/api/manifest.go`)
- Different key references = different keys (egress resolves per service def)
- SIGHUP atomic reload on enforcers
- `agency admin rebuild <agent>` to regenerate all derived config files

## Migration

Presets without `scopes` declarations work exactly as today — one key, all
tools, no scope validation. `agency admin doctor` flags these agents as
"unscoped — all tools available, no enforcer scope checks."

Operators who don't need multi-key keep using one service definition with
one key. The scope declarations still provide documentation value — doctor
shows what the agent actually needs even if the operator uses a single
broad key.

Adding `scope` to a tool definition enables enforcer validation for that
tool. Tools without `scope` fields pass through unchecked (backward
compatible). Migration is incremental — add scope annotations one service
at a time.
