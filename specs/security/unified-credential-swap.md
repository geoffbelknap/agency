# Unified Credential Swap

Last updated: 2026-04-01

Consolidate credential injection into a single, extensible, ASK-compliant system at the egress proxy boundary. All credentials are stored in the encrypted credential store and resolved at runtime via the gateway's Unix socket API.

## Architecture

All outbound requests from the agency environment flow through the egress proxy. All credential injection happens there. No other container holds or resolves secret API keys.

```
Credential Store                Gateway                    Egress Proxy
(~/.agency/credentials/         (serves Unix socket        (resolves credentials
 store.enc, AES-256-GCM)        at ~/.agency/run/           via SocketKeyResolver,
                                 gateway.sock)               injects into requests)
        │                             │                           │
        └── credstore.Put() ──────────┤                           │
                                      ├── GenerateSwapConfig() ──→│ credential-swaps.yaml
                                      ├── /internal/resolve ─────→│ real key values
                                      │                           │
                                      │                    request arrives
                                      │                    → match domain or X-Agency-Service
                                      │                    → resolve credential via socket
                                      │                    → dispatch to handler by type
                                      │                    → inject credential
                                      │                    → forward request
```

### Key Components

| Component | Role |
|-----------|------|
| **Credential store** (`~/.agency/credentials/store.enc`) | Single source of truth for all credentials. AES-256-GCM encrypted. Managed via `agency creds set/list/show/delete/rotate/test`. |
| **Gateway Unix socket** (`~/.agency/run/gateway.sock`) | Internal API for credential resolution. Egress calls `SocketKeyResolver` to fetch real credential values at runtime. |
| **credential-swaps.yaml** (`~/.agency/infrastructure/credential-swaps.yaml`) | Generated routing table. Maps domains and service names to credential references and protocol handlers. Derived from the credential store — never edited directly. |
| **Enforcer** | Scope checking only (`CheckScope`). Passes `X-Agency-Service` header through to egress. Never holds real keys. |
| **Egress proxy** | The credential injection boundary. Resolves credentials via `SocketKeyResolver`, dispatches to typed handlers, injects into outbound requests. |

## Principles

1. **One path out.** All outbound requests go through the egress proxy. All credential injection happens there. No other container holds or resolves secret API keys.
2. **One store.** The encrypted credential store (`store.enc`) holds all credentials with metadata. No `.service-keys.env`, no secrets in `.env`.
3. **Extensible by type.** Adding a new authentication mechanism means writing a handler function and registering it. No structural changes.
4. **Vault-ready.** Key resolution is behind the `SecretBackend` interface. Today it uses AES-256-GCM file encryption. Tomorrow it can use HashiCorp Vault. Swap handlers and config don't change.

## Design

### Credential Store

All credentials live in the encrypted store with rich metadata:

```bash
# Store a provider credential
agency creds set --name PROVIDER_A_API_KEY --kind provider \
  --scope platform --protocol api-key \
  --header Authorization --format "Bearer {key}" \
  --domains provider-a.example.com \
  --value sk-provider-a-...

# Store a service credential with JWT exchange (inherits group config)
agency creds set --name LC_API_KEY --kind service \
  --scope agent:detection-engineer \
  --service limacharlie-detection-engineer \
  --group limacharlie \
  --value <api-key>

# List credentials (values always redacted)
agency creds list

# Test credential against external API
agency creds test PROVIDER_A_API_KEY
```

Each credential entry includes metadata: `kind`, `scope`, `protocol`, `protocol_config`, `source`, `group`, `requires`, `external_scopes`, timestamps. See `credential-architecture.md` for the full schema.

### Generated Config: `credential-swaps.yaml`

Location: `~/.agency/infrastructure/credential-swaps.yaml`

Generated by the gateway from the credential store. Operators do not edit this file directly — it is regenerated on `agency creds set`, `agency creds delete`, `hub upgrade`, `hub activate`, `hub deactivate`, and `agency setup`.

```yaml
swaps:
  provider-a:
    type: api-key
    domains: [provider-a.example.com]
    header: Authorization
    format: "Bearer {key}"
    key_ref: PROVIDER_A_API_KEY

  nextdns-api:
    type: api-key
    domains: [api.nextdns.io]
    header: X-Api-Key
    key_ref: nextdns-api

  limacharlie-api:
    type: jwt-exchange
    domains: [api.limacharlie.io]
    token_url: "https://jwt.limacharlie.io"
    token_params:
      oid: "${LC_ORG_ID}"
      secret: "${credential}"
    token_response_field: jwt
    token_ttl_seconds: 3000
    inject_header: Authorization
    inject_format: "Bearer {token}"
    key_ref: limacharlie-api

  github:
    type: github-app
    domains: [api.github.com]
    app_id_ref: GITHUB_APP_ID
    private_key_path: /app/secrets/github-app/private-key.pem
    installation_id_ref: GITHUB_INSTALLATION_ID
```

Fields common to all types:
- `type` — dispatches to handler (`api-key`, `jwt-exchange`, `github-app`)
- `domains` — list of domains to match against request destination
- `key_ref` — name used to resolve the real key from the credential store via the gateway socket

The file is mounted read-only into the egress container. Reloaded on SIGHUP.

### Operator Overrides: `credential-swaps.local.yaml`

Same location, same format. Merged on top of the generated file. Entries here take precedence on domain conflict.

```yaml
swaps:
  internal-api:
    type: api-key
    domains: [api.internal.corp.com]
    header: Authorization
    format: "Bearer {key}"
    key_ref: INTERNAL_API_KEY
```

Load order:
1. `credential-swaps.yaml` (generated)
2. `credential-swaps.local.yaml` (operator overrides, merged on top)

Domain conflicts: local wins, logged as a warning.

Never overwritten by `hub upgrade` or `agency setup`.

### Key Resolution: SocketKeyResolver

The egress proxy resolves `key_ref` values through the `SocketKeyResolver`, which calls the gateway's internal credential resolve API over the Unix socket at `~/.agency/run/gateway.sock`:

```
GET /internal/credentials/resolve?name=PROVIDER_A_API_KEY
-> {"value": "sk-provider-a-...", "protocol": "api-key", "protocol_config": {...}}
```

The socket is mounted read-only into the egress container. Real credentials never touch the filesystem inside any container — they are resolved in-memory at request time.

Future: `VaultKeyResolver` reads from HashiCorp Vault KV. Swap handlers and config are unchanged. Vault integration must not introduce a path where non-egress containers can query Vault — the resolver runs only inside the egress process.

### Swap Handler Dispatch

The egress proxy's `request()` hook:

```
request arrives
  → extract destination domain
  → check domain is in egress allowlist (defense-in-depth)
  → look up domain in swap registry (built from credential-swaps.yaml)
  → if match: dispatch to handler by type
  → handler resolves key via SocketKeyResolver and injects credentials
  → forward request
```

Handler types:

| Type | Behavior |
|------|----------|
| `api-key` | Resolve `key_ref` via socket, inject as `header` with optional `format` (default: raw key) |
| `jwt-exchange` | Resolve `key_ref` via socket, exchange for token at `token_url`, cache with TTL, inject `inject_header` with `inject_format` |
| `github-app` | Load private key, generate JWT, exchange for installation token, inject `Authorization: token {value}` |

Each handler is a function: `(flow, swap_config, key_resolver) → None`.

Dispatch is a dict: `{"api-key": handle_api_key, "jwt-exchange": handle_jwt_exchange, "github-app": handle_github_app}`.

Adding a new type: write a handler function, add to dispatch dict, document config fields.

### Agent Service Calls (X-Agency-Service)

The `X-Agency-Service` header path for agent tool calls:

1. Enforcer validates scope (`CheckScope`) and passes the header through
2. Egress reads service name from header
3. Egress removes the header (not sent upstream)
4. Egress looks up the service name in the swap registry
5. Egress dispatches to the handler, resolving credentials via SocketKeyResolver

This unifies the agent service call path with the infrastructure credential path — same handlers, same credential store, same config.

### Credential Injection Guard: Egress Allowlist Check

Defense-in-depth: the egress proxy only injects credentials for domains that appear in the egress allowlist. Even if a swap entry were created for a malicious domain, the egress allowlist would block the request before credential injection occurs.

Check order:
1. Is destination domain in egress allowlist? If no → block request, no credential injection.
2. Is destination domain in swap registry? If yes → inject credentials.
3. Forward request.

### Routing.yaml Cleanup

Credential fields removed from `routing.yaml`:
- `auth_env` — removed
- `auth_header` — removed
- `auth_prefix` — removed

`routing.yaml` retains: `api_base`, models, pricing, rate limits. It is purely about routing decisions.

The gateway generates `credential-swaps.yaml` entries for providers by reading `routing.yaml` provider domains and matching them to credentials in the credential store.

### Intake Poller Changes

- **Does not resolve `type: secret` credentials.** `Template(header).safe_substitute(env)` continues for `type: config` / `scope: env-var` values only (e.g., `NEXTDNS_PROFILE_ID` in URLs).
- **Connector YAMLs must not put secrets in headers.** If a connector has `scope: service-grant` credentials, auth is handled by the egress proxy via domain matching. No `headers:` block for secrets.
- **Requests continue through `HTTPS_PROXY` to egress.** No change to the transport path.
- **Egress health gate.** The poll loop and schedule loop check egress proxy reachability before each tick. If egress is unreachable, the tick is skipped with a single warning log.

### Activation Validation

When a connector is activated, if it declares a `scope: service-grant` credential:
1. Verify a matching swap entry exists (or will be generated) for the connector's egress domains.
2. Verify the `key_ref` has a value in the credential store.
3. Fail activation with a clear error if either check fails.

### Hub Integration

When `hub upgrade` or `hub activate` runs:
1. Service definition YAMLs are synced to `~/.agency/registry/services/`.
2. The gateway reads all active service definitions and `routing.yaml` provider entries.
3. `credential-swaps.yaml` is regenerated from the credential store.
4. The egress proxy receives SIGHUP to reload.

Source of truth for "what services exist" is the service definition YAMLs. `credential-swaps.yaml` is compiled output derived from the credential store.

Regeneration triggers: `hub upgrade`, `hub activate`, `hub deactivate`, `agency setup`, `agency creds set`, `agency creds delete`.

### Audit Requirements

From ASK review:

1. **Swap match logging.** Every credential injection is logged: timestamp, domain, swap name, type. The real key value is never logged.
2. **Config reload logging.** SIGHUP reload produces an audit entry: "credential swap config reloaded at T, N entries, M changed since last load."
3. **Swap generation validation.** During `credential-swaps.yaml` generation, entries are only created from trusted sources (service definitions in `~/.agency/registry/services/` and provider entries in `routing.yaml`). Arbitrary connector YAML fields cannot create swap entries.

## Scope

**In scope:**
- Credential store (`store.enc`) as single source of truth
- `credential-swaps.yaml` generated routing table from credential store
- `credential-swaps.local.yaml` operator overrides
- SocketKeyResolver for egress credential resolution via gateway Unix socket
- Enforcer scope checking only (CheckScope), no credential holding
- Swap handler dispatch: `api-key`, `jwt-exchange`, `github-app` types
- Key resolution abstraction for future Vault backend
- Remove auth fields from `routing.yaml`
- Intake poller: does not resolve `type: secret` credentials
- Activation validation: verify swap entry and credential exist
- Swap config regeneration on hub upgrade/activate/deactivate/setup/creds set/delete
- Egress allowlist check before credential injection
- Audit logging: swap matches, config reloads, generation validation

**Not in scope:**
- Vault integration (design accommodates it via SecretBackend interface)
- OAuth2 client credentials swap type (add when needed)
- MTLS swap type (add when needed)
- Credential auto-rotation policies
