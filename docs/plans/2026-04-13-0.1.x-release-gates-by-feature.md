## Purpose

This document defines the concrete release gates for `0.1.x`.

Each feature area is classified as:

- `Blocking` — must pass before tagging `0.1.x`
- `Soft-blocking` — should pass; may be waived only with an explicit decision
- `Non-blocking` — important, but not a release stopper for friendly alpha

Each gate lists:

- why it matters
- automated validation
- live smoke validation
- current status / known blocker

## Release Gate Table

| Feature | Gate Level | Why It Matters | Current Status |
|---------|------------|----------------|----------------|
| Agent runtime core | `Blocking` | If agents cannot be created, started, or run reliably, nothing else matters. | Generally healthy; must be revalidated on current `main` before tag. |
| Dynamic agent reconfiguration | `Blocking` | Prompt/identity changes are a core operator capability and must affect live agent behavior, not just files on disk. | Live config update + DM verification now works and is part of the fundamentals gate. |
| Comms / DM flow | `Blocking` | Research use depends on sending a task and getting a reply back. | Strong; already validated repeatedly. |
| Routing / provider execution | `Blocking` | Agents must actually reach an LLM provider and return a response. | Healthy, but should be gated with a real provider-backed task. |
| Budget + usage + audit linkage | `Blocking` | The platform claims auditable economics; release should prove that. | Pass on basic fidelity; still worth keeping as a release gate. |
| Graph query / stats | `Blocking` | Knowledge retrieval is part of the product and should not silently regress. | Query/stats work. |
| Graph ingestion | `Soft-blocking` | Ingestion is a promised capability, but may not be required for first friendly tester workflows. | Pass for basic local ingest; still worth keeping as a gated surface. |
| Team mission routing / failover / deconfliction | `Soft-blocking` | Team assignment is part of the platform model and should work for the implemented alpha subset. | Dedicated live gate now passes for coordinator routing, no-coordinator claims, and coverage failover. |
| Drive authority runtime (read + consent-gated mutation) | `Soft-blocking` | Drive administration is part of the platform story and the implemented alpha subset should work end to end. | Dedicated live gate now passes for read, consent denial, and consent-authorized mutation. |
| Trajectory monitoring visibility | `Soft-blocking` | The platform claims agentic oversight and anomaly visibility. | Fundamentals gate now passes live trajectory checks; remaining gap is better product surfacing, not endpoint availability. |
| Web UI core non-destructive flows | `Blocking` | Friendly testers will use the UI; it must be coherent and safe. | Live suites green. |
| Web UI destructive flows | `Non-blocking` | Important later, but intentionally excluded from this alpha pass. | Not part of current gate. |
| Relay-hosted access | `Soft-blocking` | If relay is part of tester onboarding, it must work. | Working, but still has polish debt. |
| Homebrew install path | `Blocking` | `0.1.x` is supposed to be installable by testers. | Workflow exists; needs post-tag smoke. |
| GHCR image publish path | `Blocking` | `agency infra up` in release mode depends on it. | Current tag has visibility issues; future publish fix is in `#188`. |

## Suggested Blocking Set For `0.1.x`

If the goal is a friendly alpha release, I would block the tag on:

- agent runtime core
- comms / DM flow
- routing / provider execution
- budget + usage + audit linkage
- graph query / stats
- web UI core non-destructive flows
- Homebrew install path
- GHCR image publish path

I would treat these as soft-blocking but very important:

- graph ingestion
- trajectory monitoring visibility
- relay-hosted tester flow
