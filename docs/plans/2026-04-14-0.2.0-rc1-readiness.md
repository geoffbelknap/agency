## Purpose

This document captures the current evidence for deciding whether to tag
`v0.2.0-rc1`.

The question is narrower than "is `0.2.0` fully done?" The question is:

- is the intentionally scoped core product coherent?
- do the local core readiness gates pass?
- are the remaining unknowns mainly release-publish validation rather than core
  product correctness?

## Current Read

Current recommendation: **close to ready to tag `v0.2.0-rc1`, but not yet ready
to call `0.2.0` done.**

Why:

- the core product story is now coherent across docs, CLI, web, OpenAPI, and
  MCP
- the local core-path readiness checks passed
- the remaining unknowns are concentrated in published release/installability
  validation, not in the narrowed product contract itself

- README and top-level docs rewritten around the core product

- experimental pages now self-identify as experimental

## What Still Lacks Evidence

These are the main reasons not to skip directly to `0.2.0` final yet:

### 1. Published Release Validation

Not yet evidenced in this pass:

- tagged GitHub release artifact validation
- post-tag Homebrew install validation
- post-tag GHCR image visibility/pullability validation
- `./scripts/release-readiness-check.sh published --tag v0.2.0-rc1`

This is the biggest remaining gap.

### 2. Clean-Machine Install Evidence

Local setup flow is strong, but this pass did not yet prove:

- one clean-machine Homebrew install
- one clean-machine end-to-end core smoke on the tagged artifact

### 3. Release Workflow Observation

This pass did not yet verify the actual `rc1` publish workflows end to end:

- GitHub release workflow behavior on `v0.2.0-rc1`
- image publish workflow behavior on `v0.2.0-rc1`
- any version-string or formula/tap edge cases on the prerelease tag

## `rc1` Decision Standard

Reasonable standard for tagging `v0.2.0-rc1`:

- core contract is coherent
- local core-path gates pass
- no known severe runtime regression remains in the narrowed core
- remaining uncertainty is mostly in publish/install validation

Reasonable standard for tagging `v0.2.0` final:

- all of the above
- published release checks pass
- Homebrew path verified
- GHCR path verified
- clean-machine smoke verified

## Recommendation

If the goal is to learn quickly from a prerelease tag, the repo now looks
**credible for a `v0.2.0-rc1` decision**.

If the goal is to cut `v0.2.0` final immediately, the repo is **not there
yet**, because published release/install evidence is still missing.

## Suggested Next Steps

1. Tag `v0.2.0-rc1`.
2. Watch the GitHub release and image publish workflows closely.
3. Run `./scripts/release-readiness-check.sh published --tag v0.2.0-rc1`.
4. Perform one clean-machine Homebrew install smoke.
5. Perform one clean-machine core product smoke on the released artifact.
6. If those pass, either:
   - promote toward `v0.2.0`, or
   - cut `v0.2.0-rc2` only if publish/install issues appear.
