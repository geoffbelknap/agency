## Purpose

This document captures the current evidence for deciding whether to tag
`v0.2.0-rc1`.

The question is narrower than "is `0.2.0` fully done?" The question is:

- is the intentionally scoped core product coherent?
- do the local core readiness gates pass?
- are the remaining unknowns mainly release-publish validation rather than core
  product correctness?

## Current Read

Current recommendation after the actual `rc1` publish: **use `v0.2.0-rc1` as
the proving release, fix the image-pipeline issue, and prepare `v0.2.0-rc2`.**

Why:

- the core product story is now coherent across docs, CLI, web, OpenAPI, and
  MCP
- the local core-path readiness checks passed
- the GitHub release artifacts and Homebrew publish path succeeded on `rc1`
- the remaining release blocker is concentrated in the container-image publish
  workflow

## Remaining Blockers Before `0.2.0` Final

### 1. Container image publish failure

Observed on `v0.2.0-rc1`:

- `agency-python-base` published successfully
- `embeddings`, `web`, `egress`, `workspace`, `enforcer`, and `web-fetch`
  published successfully
- `comms`, `knowledge`, `intake`, and `body` failed

Root cause:

- those repo-context images used `FROM agency-python-base:latest`
- in GitHub Actions that resolved to
  `docker.io/library/agency-python-base:latest`
- the intended base image lives in GHCR, not Docker Hub

Planned `rc2` fix:

- use a configurable `PYTHON_BASE_IMAGE`
- pass the exact GHCR tag from the workflow

### 2. Published validation rerun after the image fix

Still needed after `rc2`:

- `./scripts/release-readiness-check.sh published --tag v0.2.0-rc2`
- confirm GHCR image manifests and pullability for the supported stack

### 3. Clean-Machine Install Evidence

Local setup flow is strong, but this pass did not yet prove:

- one clean-machine Homebrew install
- one clean-machine end-to-end core smoke on the tagged artifact

## `rc1` Decision Standard

Reasonable standard for tagging `v0.2.0-rc1`:

- core contract is coherent
- local core-path gates pass
- no known severe runtime regression remains in the narrowed core
- remaining uncertainty is mostly in publish/install validation

Reasonable standard for tagging `v0.2.0` final:

- all of the above
- published release checks pass
- Homebrew path verified
- GHCR path verified
- clean-machine smoke verified

## Recommendation

`v0.2.0-rc1` served its purpose: it proved the narrowed release line is real
enough to publish and exposed a concrete container-image pipeline bug.

Do not promote `rc1` toward final.

Fix the image-base reference bug, cut `v0.2.0-rc2`, and rerun published
validation from there.
