## Purpose

This document defines the concrete release gates for the first intentionally
scoped `0.2.x` Agency line.

This is not a broad "everything in the repo" release. It is the narrower
`core` Agency product:

- governed single-agent runtime
- direct-message workflow
- event-driven execution
- trimmed graph retrieval/context
- auditable provider-backed work
- core web/API/MCP surfaces

Each feature area is classified as:

- `Blocking` — must pass before tagging `0.2.x`
- `Soft-blocking` — should pass; may be waived only with an explicit decision
- `Non-blocking` — important for ongoing development, but not part of the core
  release contract

## Release Gate Table

| Feature | Gate Level | Why It Matters | Current Stance |
|---------|------------|----------------|----------------|
| Install and setup path | `Blocking` | If a tester cannot install the binary, configure one provider, and bring the stack up, the product is not shippable. | Core requirement for `0.2.x`. |
| Agent runtime core | `Blocking` | Create/start/show/stop must be reliable or the core story fails immediately. | Remains the strongest product surface. |
| Dynamic agent reconfiguration | `Blocking` | Updating a running agent is part of the supported operator path, not a side demo. | Keep this in the mainline release bar. |
| DM / comms core | `Blocking` | First-user success depends on sending work to an agent and receiving a reply. | Primary interaction model. |
| Event-driven runtime delivery | `Blocking` | Agency should react through events and subscriptions, not rely on polling as the default contract. | Core architectural commitment. |
| WebSocket/realtime delivery | `Soft-blocking` | Important for chat feel and operator visibility, but secondary to the event model itself. | Keep honest but not dominant. |
| Provider execution and basic routing | `Blocking` | Agents must reach a real provider and execute predictable work. | Core path. |
| Budget, usage, and audit linkage | `Blocking` | Governed execution requires visible economics and durable evidence. | Core claim, not optional polish. |
| Graph retrieval/query/stats core | `Blocking` | The `0.2.x` story includes agents improving over time through durable graph-backed context. | Narrow graph core only. |
| Graph write/update path for useful knowledge compounding | `Soft-blocking` | The graph story should be more than read-only, but the release should not be blocked by the broader ingestion/governance sprawl. | Keep scope narrow. |
| Core web UI surfaces | `Blocking` | Setup, agents, DM, and basic activity must match the supported core product. | Required for first users. |
| Core REST/OpenAPI contract | `Blocking` | Builders and internal clients need a truthful supported contract. | `/api/v1/openapi-core.yaml` is part of the bar. |
| Core MCP discovery and tools | `Blocking` | MCP is now a supported operator/programmatic surface and must default to the core contract. | Must stay aligned with API tiering. |
| Thin SDK/client path | `Soft-blocking` | The API should be buildable against, but a large multi-language SDK story is not required for the first `0.2.x` tag. | Keep one honest client path. |
| Release/publish path | `Blocking` | Homebrew and GHCR are part of the install story and must work for the supported path. | Required for a real release. |
| Experimental surfaces remain hidden by default | `Blocking` | The product line resets only if non-core surfaces stop masquerading as equal peers. | Docs, CLI, web, OpenAPI, and MCP must stay aligned. |
| Missions, teams, coordinators | `Non-blocking` | Worth continuing, but no longer part of the default contract. | Experimental only. |
| Packs, hub, instances, connector breadth | `Non-blocking` | Ecosystem/platform work, not core first-user success. | Experimental only. |
| Graph governance, ontology, review/quarantine | `Non-blocking` | Important graph support machinery, but not part of the trimmed core. | Experimental/internal only. |
| Routing optimizer / trust calibration | `Non-blocking` | Basic routing matters; optimization workflows do not block `0.2.x`. | Experimental only. |
| Relay-hosted path | `Non-blocking` | Optional distribution path, not part of the default local core release. | Revisit separately. |

## Gate Definitions

### 1. Install And Setup Path

**Gate level:** `Blocking`

**Pass means:**

- the release artifact installs cleanly
- one provider can be configured through the supported setup path
- `agency infra up` succeeds for the supported local stack
- `agency admin doctor` does not report critical safety or deployment issues

**Primary validation:**

- `/Users/geoffbelknap/Documents/GitHub/agency-workspace/agency/scripts/setup-wizard-readiness-check.sh`

## Suggested Blocking Set For `0.2.x`

Treat these as the hard no-waiver set unless there is an explicit product
decision otherwise:

- install and setup path
- agent runtime core
- dynamic agent reconfiguration
- DM/comms core
- event-driven runtime delivery
- provider execution and basic routing
- budget/usage/audit linkage
- graph query/retrieval/stats core
- core web UI surfaces
- core API/OpenAPI/MCP contract
- release/publish path
- experimental surfaces hidden by default

## Explicit Non-Blocking Set For `0.2.x`

These can keep shipping as `experimental` work without blocking the first core
release line:

- missions, teams, and coordinator workflows
- packs, hub lifecycle, packages, and instances
- connector breadth, Slack, and Drive productized flows
- graph governance and ontology management
- routing optimizer workflows
- relay-hosted product path

## Working Principle

`0.2.x` should be the first version where the product contract is smaller,
clearer, and more honest than the repo surface.
