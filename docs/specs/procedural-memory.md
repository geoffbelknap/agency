The knowledge graph stores organizational facts — people, systems, decisions, incidents, lessons — but not how-to knowledge. An agent that triaged 100 tickets last week has no memory of which approaches worked, which tools were most effective, or what mistakes to avoid. Each new task is approached as if it were the first.

Procedural memory adds a layer that captures task-execution patterns from completed tasks and surfaces relevant patterns at the start of new tasks. It builds on the existing knowledge graph ontology and the mission knowledge injection mechanism already in the system prompt assembly pipeline.


## Problem

Agents operate statelessly across tasks. The body runtime maintains persistent working memory at `/workspace/.memory/`, and the knowledge graph provides mission-scoped organizational context at session start. Neither captures procedural knowledge: the specific sequences of actions, tool selections, and decision rationale that led to successful (or failed) task outcomes.

This means:
- Agents cannot learn from repetition. The 50th ticket triage runs with the same zero-knowledge approach as the 1st.
- Successful approaches are not preserved. An agent that discovers an effective workflow forgets it at task boundary.
- Failed approaches are not avoided. Without records of what did not work, agents repeat the same mistakes.
- Cross-agent knowledge transfer requires manual operator effort. Two agents on the same mission cannot benefit from each other's experience.


## Design Overview

After each task completion, the body runtime generates a structured **procedure record** summarizing what was done, which tools were used, and whether it worked. The record is contributed to the knowledge graph as a new `procedure` entity type. At the start of each new task, the body runtime queries for relevant past procedures and injects them into the system prompt alongside existing mission knowledge.

The mechanism reuses existing infrastructure:
- Knowledge graph for storage (via `contribute_knowledge`)
- Mission knowledge injection for retrieval (extended with a new prompt block)
- Enforcer mediation for all LLM calls and knowledge graph access
- `task_complete` signal as the capture trigger


## Procedure Record

A procedure record captures what happened during a single task execution. It is generated by a lightweight LLM summarization call after the agent signals `task_complete`.

```yaml
entity_type: procedure
attributes:
  agent: henrybot900
  mission_id: 8a3f2b1c-4d5e-6f7a-8b9c-0d1e2f3a4b5c
  mission_name: ticket-triage
  task_type: incident_triage        # derived from mission name or explicit tag
  task_id: task-abc123
  timestamp: 2026-03-27T10:00:00Z
  approach: |
    1. Read ticket details from Jira
    2. Checked service dependency graph for impact scope
    3. Cross-referenced with recent incident history
    4. Assigned P2 based on limited blast radius
    5. Tagged @oncall-infra and posted to #incidents
  tools_used: [jira_get_issue, query_knowledge, channel_send]
  outcome: success                  # success | partial | failed
  duration_minutes: 4
  reflection_notes: ""              # from reflection loop, if enabled
  lessons: []                       # specific things learned
```

### Field Details

- **`agent`** — Name of the agent that executed the task. Immutable after creation.
- **`mission_id`** — UUID of the assigned mission. Links procedure to mission for scoped retrieval. Required.
- **`mission_name`** — Human-readable mission name at time of capture. Informational only — queries use `mission_id`.
- **`task_type`** — Categorization tag derived from the mission name or set explicitly by the agent. Used for cross-mission procedure discovery in future work.
- **`task_id`** — The task identifier from the `task_complete` signal. Links procedure to the specific task execution.
- **`timestamp`** — ISO 8601. When the procedure was captured.
- **`approach`** — Natural language description of what the agent did, in sequential steps. This is the core of the record — it tells future runs what worked (or did not).
- **`tools_used`** — List of tool names invoked during the task. Enables tool-based filtering (e.g., "show me procedures that used the Jira connector").
- **`outcome`** — `success` (task completed as expected), `partial` (task completed with caveats or degraded result), `failed` (task did not achieve its goal). Set by the summarization call based on conversation context.
- **`duration_minutes`** — Wall-clock time from task start to `task_complete` signal.
- **`reflection_notes`** — Output from the reflection loop, if enabled for this agent. Empty string if reflection is disabled.
- **`lessons`** — Specific takeaways the agent identified. These are extracted from the conversation — things like "the Jira API returns paginated results, need to check `nextPage`" or "P1 tickets should always cc the VP-Eng channel."


## Capture Mechanism

Procedure capture fires after the `task_complete` signal. The body runtime initiates a separate one-shot LLM call through the enforcer — this is not part of the agent's main conversation.

### Capture Flow

```
1. Agent calls complete_task(summary=...)
2. Body runtime emits task_complete signal via comms
3. Body runtime assembles capture prompt:
   - Conversation history (summarized using the same LLM-based summarization that context management uses — not raw truncation)
   - Task metadata (mission_id, task_id, start time, tools invoked)
   - Structured output schema for the procedure record
4. Body runtime sends capture prompt through enforcer LLM proxy
5. LLM returns structured procedure record
6. Body runtime calls contribute_knowledge(entity_type="procedure", ...)
7. Knowledge graph stores the record
```

The capture prompt instructs the LLM to extract the procedure record from the conversation. It asks for steps taken (not planned), actual tools used (not available), and honest outcome assessment. The prompt is a platform prompt — not influenced by agent identity or mission instructions.

### Budget

The capture LLM call is a platform cost. It is not charged to the agent's task budget because it runs after `task_complete` — the task's budget accounting is already closed. A platform-reserved budget covers procedure and episode generation. The platform budget is configured at the gateway level (`platform_budget` in gateway config) and tracked separately from agent budgets. The default is generous enough for lightweight summarization calls — operators can adjust based on model pricing and task volume.

The capture call uses the same egress-mediated LLM path as platform-side evaluation (see Mission Success Criteria spec). The gateway sends the request directly to the egress proxy with a platform-level credential.


## Ontology Addition

The `procedure` entity type is added to the knowledge graph ontology alongside existing types (person, system, decision, finding, incident, lesson, process, metric, goal, narrative, pattern, preference).

```yaml
entity_types:
  procedure:
    description: A task-execution pattern recording how a specific task was accomplished
    attributes:
      - agent
      - mission_id
      - mission_name
      - task_type
      - task_id
      - timestamp
      - approach
      - tools_used
      - outcome
      - duration_minutes
      - reflection_notes
      - lessons
```

### Relationships

```
procedure --instance_of--> mission
procedure --performed_by--> agent (entity in knowledge graph, if present)
```

The `instance_of` relationship links every procedure to its parent mission. This is the primary retrieval axis — "give me procedures for mission X." The `performed_by` relationship is optional and created only if the agent exists as an entity in the knowledge graph.


## Retrieval and Injection

At session start, during system prompt assembly, the body runtime queries the knowledge graph for procedures matching the current mission. This happens after mission knowledge injection and before general memory.

### Query

```
entity_type: procedure
filter: mission_id = {current_mission_id} AND outcome IN (success, partial)
sort: timestamp DESC
limit: {max_retrieved}  # default: 5
```

The query returns the most recent successful and partial procedures by default. Failed procedures are excluded from prompt injection to avoid biasing the agent toward failure avoidance over task completion.

When `include_failures` is `true` in the configuration, the query also returns the most recent failed procedures (up to 2). Failed procedures are injected with explicit framing:

```
### Failed Procedure from 2026-03-24 (8 min, failed)
⚠ This approach did NOT work. Avoid repeating it.
1. Attempted direct Jira API call without checking rate limits
2. Hit 429 rate limit after 3 rapid queries
3. Retried immediately without backoff — locked out for 60s
Lesson: Always check rate limit headers before batch queries
```

This option is useful for missions where avoiding known failure modes is more valuable than the risk of anchoring on failures. Default: `false`. Recommended to enable only after the agent has accumulated enough successful procedures to provide positive reference alongside the failures.

### Prompt Block

Retrieved procedures are formatted as a new block in the system prompt:

```
## Relevant Past Procedures

Based on your previous experience with this mission, here are approaches that worked:

### Procedure from 2026-03-26 (4 min, success)
1. Read ticket details from Jira
2. Checked service dependency graph for impact scope
3. Cross-referenced with recent incident history
4. Assigned P2 based on limited blast radius
5. Tagged @oncall-infra and posted to #incidents

### Procedure from 2026-03-25 (6 min, success)
1. ...

Use these as reference — adapt to the current situation, don't follow blindly.
```

The closing instruction ("adapt to the current situation") is critical. Procedures are reference material, not scripts. The agent retains full autonomy over how to approach the current task.

### XPIA Scanning

Procedure `approach` and `lessons` fields are LLM-generated content from past agent conversations. If a past task involved processing adversarial input (e.g., a malicious ticket body), the procedure record could contain adversarial content that gets re-injected into future system prompts. This is a prompt injection vector.

All procedure content MUST pass through XPIA scanning before injection into the system prompt. The body runtime submits the assembled procedural memory block to the enforcer's XPIA scanner (the same path used for tool-role messages) before inserting it into the prompt. If the scanner flags content, the flagged procedure is excluded from injection and a warning signal (`procedure_xpia_flagged`) is emitted. The procedure record remains in the knowledge graph but is quarantined from prompt injection until an operator reviews it.

This applies to all content entering the system prompt from the knowledge graph — procedures, episodes, and mission knowledge alike.

### Prompt Assembly Order

The procedural memory block is inserted between mission knowledge and general memory:

```
identity.md
mission instructions
mission behavioral frame
mission knowledge          ← existing: org facts scoped to mission
procedural memory          ← NEW: past approaches scoped to mission
memory index               ← agent's private working memory
comms context
FRAMEWORK.md
AGENTS.md
skills
```

This placement ensures procedures are available alongside mission context but before the agent's own memory state. The agent sees "here's what your mission is" (instructions), "here's what the organization knows about this domain" (mission knowledge), and "here's what has worked before" (procedural memory) as a coherent briefing.


## Procedure Pruning

Procedures accumulate over time. Without pruning, a long-running mission would generate hundreds of records, most of which add no incremental value. The gateway runs a consolidation process to keep the procedure set useful and bounded.

### Retention Rules

- Keep the **20 most recent successful** procedures per mission
- Keep **all failed procedures** up to 10 per mission (failures are high-signal)
- Partial procedures follow the success retention rules (20 most recent)
- Older records beyond these limits are deleted from the knowledge graph

### Consolidation

When a mission accumulates more than 50 procedures (across all outcomes), the gateway triggers a consolidation pass:

1. Query all successful procedures for the mission
2. Send them to a one-shot LLM call with a consolidation prompt
3. The LLM produces a single **consolidated procedure** — a merged reference document capturing the most common patterns, notable variations, and key lessons
4. The consolidated procedure is stored as a new `procedure` entity with `task_type: consolidated` and `outcome: success`
5. The original procedures that were consolidated are deleted

The consolidated procedure replaces the individual records in future retrieval. It is always the first procedure returned, followed by recent individual procedures that post-date the consolidation.

Consolidation runs as a background task in the gateway, triggered by `task_complete` when the procedure count exceeds the threshold. The LLM call uses the same platform budget as procedure capture.


## Privacy and Authorization

Procedures are organizational knowledge (ASK tenet 23), not agent-private state. Access follows the same authorization model as all other knowledge graph entities (ASK tenet 24).

### Access Scoping

| Accessor | Scope | Rationale |
|----------|-------|-----------|
| The agent that created the procedure | Full read access | Own execution history |
| Other agents on the same mission | Read access via `mission_id` | Team members benefit from shared experience |
| Agents on different missions | No access by default | Cross-mission access requires explicit authorization grant |
| Operator | Full read access to all procedures | Operator oversight |

Cross-team procedure access (e.g., an agent querying procedures from a mission it is not assigned to) requires an explicit knowledge graph authorization grant from the operator. This prevents information leakage between organizational boundaries.

### Immutability

Procedures are immutable after contribution. The agent cannot modify its own procedure records — once written to the knowledge graph via `contribute_knowledge`, the record is final. This prevents agents from revising history to appear more successful than they were. Only the pruning and consolidation processes (run by the gateway, not agents) modify procedure records.

| Command | Description |
|---------|-------------|
| `agency mission knowledge <name>` | Extended: includes procedure count and most recent procedure summary |

New purge command for rollback (ASK tenet 25 — identity mutations are recoverable):

| Command | Description |
|---------|-------------|
| `agency mission procedures purge <mission> [--before <date>] [--agent <name>]` | Delete procedure records for a mission. Supports filtering by date and agent. Requires operator confirmation. |

This is the rollback mechanism if a bad procedure contaminates future sessions. The operator can purge all procedures for a mission, or surgically remove procedures from a specific time range or agent. Purged procedures are soft-deleted (marked `purged` in the knowledge graph, excluded from queries) and permanently deleted after 30 days.

The existing `agency mission knowledge` command is extended to show procedural memory alongside mission knowledge. Output includes a "Procedures" section with the total count, most recent procedure timestamp, and a one-line summary of the most recent successful procedure.

Example output:


## Configuration

Procedural memory is configured per-agent or per-mission. Agent-level config applies to all missions unless overridden at the mission level.

### Agent Config

```yaml
# In agent config (~/.agency/agents/{name}/config.yaml)
procedural_memory:
  enabled: true                   # default: true — master switch
  capture: true                   # generate procedure records on task_complete
  retrieve: true                  # inject past procedures at session start
  max_retrieved: 5                # procedures injected into system prompt
  include_failures: false         # include failed procedures in injection (default: false)
  consolidation_threshold: 50     # trigger consolidation after this many
```

### Mission Override

```yaml
# In mission YAML (~/.agency/missions/{name}.yaml)
procedural_memory:
  capture: true
  retrieve: true
  max_retrieved: 3                # override agent default for this mission
```

Mission-level settings override agent-level settings for the duration of that mission. The `enabled` flag is agent-level only — if procedural memory is disabled on the agent, mission overrides are ignored.

## Task Tier Interaction

Procedural memory has two phases with different tier gates:

- **Capture** (post-task LLM call) — activates at `standard` and `full` tiers. Even routine mission tasks are worth capturing — the memory bank compounds over time. Skipped at `minimal` (ad-hoc DMs aren't mission work and produce no reusable procedures).
- **Retrieval/injection** (system prompt addition) — activates at `full` tier only. Injecting past procedures adds ~500-1,000 tokens to the system prompt, which compounds across every LLM call in the task. Only worthwhile for complex tasks that benefit from past experience.

When `cost_mode: frugal`, both capture and retrieval are disabled. When `cost_mode: balanced`, capture is on but retrieval is on (injected at standard+ tiers). When `cost_mode: thorough`, both are on with higher `max_retrieved` (5 vs 3).

Cost attribution: the capture LLM call is tagged with `X-Agency-Cost-Source: memory_capture` and charged to the platform budget.

See the Task Tier and Cost Model spec for the full tier classification logic.


## Deferred Work

- **Cross-mission procedure discovery** — Query procedures by `task_type` across missions. Useful when a new mission covers similar work to an existing one. Requires a task-type taxonomy and cross-authorization framework.
- **Procedure quality scoring** — Rate procedures by downstream task success. If an agent references a procedure and the subsequent task succeeds, increase the procedure's quality score. Surface high-quality procedures first.
- **Negative procedure injection** — Optionally surface failed procedures with "avoid this approach" framing. Requires careful prompt engineering to avoid anchoring the agent on failure modes.
- **Procedure sharing across agents** — Explicitly share procedures between agents on different missions via operator command. Currently limited to same-mission access.
