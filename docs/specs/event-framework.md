A unified event routing layer inside the gateway that connects event sources to subscribers. All agent activation, operator notifications, and external integrations flow through the same model: sources emit events, the bus routes them by subscription, subscribers act on matching events.

The event framework replaces multiple ad-hoc event paths (connectorâ†’intakeâ†’brief, commsâ†’idle-reply, capabilityâ†’SIGHUP) with a single routing model. It also replaces the brief delivery mechanism â€” task delivery happens through channel messages and DMs routed via the event bus.

## Event Envelope

Every event in the system has the same shape regardless of source:

```json
{
  "id": "evt-a1b2c3d4",
  "source_type": "connector",
  "source_name": "jira-ops",
  "event_type": "issue_created",
  "timestamp": "2026-03-23T10:00:00Z",
  "data": {
    "key": "INC-1234",
    "summary": "Production database latency spike",
    "priority": "P1"
  },
  "metadata": {
    "channel": "incidents",
    "author": "jira-ops"
  }
}
```

### Fields

- **`id`** â€” Unique event ID. Generated by the gateway when the event enters the bus. Used for deduplication and audit. Format: `evt-\{short_uuid\}`. Channel messages reuse the comms message ID: `evt-msg-\{comms_id\}`.
- **`source_type`** â€” One of: `connector`, `channel`, `schedule`, `webhook`, `platform`.
- **`source_name`** â€” Which specific source produced the event (connector name, channel name, webhook name, schedule name, or `gateway` for platform events).
- **`event_type`** â€” What happened. Defined by the source (e.g. `issue_created`, `message`, `cron_fired`, `agent_halted`). The bus routes on this field.
- **`timestamp`** â€” ISO 8601. When the event was produced (or received, for webhooks).
- **`data`** â€” The payload. Opaque to the framework â€” interpreted by subscribers. Structure depends on source.
- **`metadata`** â€” Routing hints. Optional fields like `channel`, `author`, `mission_id` that help the bus evaluate subscriptions without parsing `data`.

The framework routes on `source_type`, `source_name`, and `event_type`. Subscribers never need to parse `data` to decide whether to receive an event.

The event bus is not a separate service. It is a routing layer inside the gateway that evaluates subscriptions against incoming events and delivers matches. No new infrastructure, no message queue, no external dependencies.

### Event Flow

1. Source produces a raw event (HTTP POST, WebSocket message, cron fire, internal state change)
2. Gateway wraps it in the standard envelope, assigns an ID, logs receipt
3. Gateway evaluates all active subscriptions against the event
4. Matching events are delivered to each subscriber (agent wake, HTTP POST, channel message)
5. Delivery is logged per subscriber

## Delivery Guarantees

Events are delivered at-most-once in v1. No persistence layer. On gateway restart: in-flight events are lost, scheduled cron fires during downtime are not replayed, webhook senders receive connection errors and should implement their own retry logic. Event replay and persistence are deferred to future work.

## Event Sources

### Connector Source

Connectors deliver events through the intake service. The framework formalizes this path: intake passes events into the bus instead of directly briefing agents.

Events from connectors carry the connector name as `source_name` and the connector-defined event type as `event_type`. The `data` field contains the raw connector payload.

```json
{
  "source_type": "connector",
  "source_name": "jira-ops",
  "event_type": "issue_created",
  "data": {"key": "INC-1234", "summary": "...", "fields": {...}}
}
```

### Channel Source

Channel messages from the comms WebSocket are wrapped as events. Today the body runtime evaluates every message against interest keywords. With the framework, the gateway filters first and only delivers matching events to subscribed agents.

```json
{
  "source_type": "channel",
  "source_name": "incidents",
  "event_type": "message",
  "data": {"content": "severity:critical â€” database latency spike"},
  "metadata": {"author": "operator", "channel": "incidents"}
}
```

Channel events include two hard-coded routing rules that bypass subscriptions:
- **Direct @mentions** â€” always delivered to the mentioned agent
- **Operator DMs** â€” always delivered to the target agent

These are not subscriptions â€” they are unconditional routing rules enforced by the bus.

### Schedule Source

A cron-like timer inside the gateway. No external cron daemon.

```json
{
  "source_type": "schedule",
  "source_name": "daily-compliance-scan",
  "event_type": "cron_fired",
  "data": {"schedule": "0 9 * * MON-FRI", "fire_time": "2026-03-24T09:00:00-07:00"}
}
```

Schedule sources are defined in mission triggers:

```yaml
triggers:
  - source: schedule
    name: daily-compliance-scan
    cron: "0 9 * * MON-FRI"
```

The gateway manages schedules internally. When a mission with schedule triggers is assigned, the gateway registers the cron expression. When paused or completed, the schedule is deactivated or removed.

Timezone is derived from the mission's `health.business_hours` field, or UTC if unspecified.

### Webhook Source

Inbound HTTP endpoint on the gateway for external systems:

```
POST /api/v1/events/webhook/{name}
Content-Type: application/json
X-Webhook-Secret: {secret}

{"deployment": "prod-v2.3", "status": "complete", "commit": "abc123"}
```

The gateway wraps the POST body as the event `data`:

```json
{
  "source_type": "webhook",
  "source_name": "deploy-notifications",
  "event_type": "deployment_complete",
  "data": {"deployment": "prod-v2.3", "status": "complete", "commit": "abc123"}
}
```

**Webhook registration is required.** Unregistered webhook names are rejected with 404. This prevents arbitrary event injection.

```
agency webhook create <name>          # creates webhook, generates secret
agency webhook list                   # list registered webhooks
agency webhook show <name>            # show URL and secret
agency webhook delete <name>          # remove webhook
```

Each webhook has:
- A unique name (used as URL path segment)
- A secret token (validated via `X-Webhook-Secret` header)
- A required `event_type` (set at registration time via `--event-type <type>`)

**Implementation requirements:** Webhook secret comparison must use constant-time comparison (e.g. `crypto/subtle.ConstantTimeCompare`). The webhook endpoint must be rate-limited (default: 60 requests per minute per webhook name). If the gateway is exposed externally for webhook reception, TLS must be configured. Secret rotation is supported via `agency webhook rotate-secret <name>` (generates new secret without changing URL). HMAC-based request signing is deferred to future work â€” static bearer token is a known limitation for v1.

**event_type at registration:** The `event_type` is required at webhook registration time (set via `agency webhook create <name> --event-type <type>`). All events from that webhook use the registered event_type. This avoids parsing the POST body for routing. If the webhook needs to emit multiple event types, register multiple webhooks.

### Platform Source

Internal gateway events exposed as first-class events in the bus. These are state changes the gateway already knows about but doesn't currently route as events:

**Agent lifecycle:**
- `agent_started`, `agent_stopped`, `agent_halted`, `agent_restarted`

**Mission lifecycle:**
- `mission_assigned`, `mission_paused`, `mission_resumed`, `mission_completed`, `mission_updated`, `mission_health_alert`

**Capability changes:**
- `capability_granted`, `capability_revoked`

**Meeseeks lifecycle:**
- `meeseeks_spawned`, `meeseeks_completed`, `meeseeks_distressed`, `meeseeks_terminated`

**Infrastructure:**
- `infra_up`, `infra_down`

```json
{
  "source_type": "platform",
  "source_name": "gateway",
  "event_type": "agent_halted",
  "data": {"agent": "henrybot900", "type": "immediate", "reason": "operator stop"}
}
```

Platform events enable missions that react to system state:

```yaml
triggers:
  - source: platform
    event_type: agent_halted
```

"When any agent is halted, investigate and report to operator."

## Subscriptions

A subscription routes events matching a pattern to a destination. Subscriptions are created implicitly by other features â€” operators don't manage them directly in v1.

**Note:** The missions spec uses `source` and `connector`/`channel` as field names in trigger definitions. These map to `source_type` and `source_name` in the event framework. The missions spec trigger schema should be updated to use event framework terminology for consistency.

### Subscription Structure

```
{
  source_type: "connector",    # match on source type (required)
  source_name: "jira-ops",     # match on specific source (optional â€” omit for any)
  event_type: "issue_created", # match on event type (optional â€” omit for any)
  match: "severity:*",         # glob pattern on event content (optional, channel source only)
  destination: {
    type: "agent",             # agent, webhook, or channel
    target: "henrybot900"      # agent name, URL, or channel name
  }
}
```

The `match` field (glob pattern) applies to channel source events only. For other source types, routing is based on `source_type`, `source_name`, and `event_type` â€” no content matching.

### Implicit Subscription Creation

**Mission triggers** create subscriptions on assignment:
```
mission "ticket-triage" assigned to henrybot900
  â†’ subscribe: source_type=connector, source_name=jira-ops, event_type=issue_created â†’ henrybot900
  â†’ subscribe: source_type=channel, source_name=incidents, match=severity:* â†’ henrybot900
```

When the mission is paused, subscriptions are deactivated (not deleted). When resumed, they reactivate. When completed, they are removed.

**Operator notifications** create outbound subscriptions from `config.yaml`:
```
notifications config has ntfy for mission_health
  â†’ subscribe: source_type=platform, event_type=mission_health â†’ POST https://ntfy.sh/my-alerts
```

### Subscription Table

The gateway maintains an in-memory subscription table derived from:
1. Active missions (their triggers)
2. Notification config (outbound webhooks)
3. Hard-coded rules (@mentions, operator DMs)

The table is rebuilt when missions change state or config is reloaded. No persistent subscription storage â€” the table is fully derivable from current state.

## Outbound Subscriptions (Notifications)

External notification forwarding is configured at the platform level:

```yaml
# ~/.agency/config.yaml
notifications:
  - name: ops-alerts
    type: ntfy
    url: https://ntfy.sh/my-agency-alerts
    events: [mission_health, agent_halted, quarantine, meeseeks_distressed]

  - name: slack-feed
    type: webhook
    url: https://hooks.slack.com/services/T00/B00/xxx
    events: [mission_health, mission_completed, agent_started]
    headers:
      Content-Type: application/json
```

The gateway matches outbound subscriptions against the event bus. When an event's `event_type` matches an entry in `events`, the gateway POSTs the event envelope to the URL.

**Delivery semantics:**
- Fire and forget with one retry on failure
- Failed deliveries are logged in the gateway audit log
- No exponential backoff, no dead letter queue (v1 simplicity)
- The POST body is the full event envelope as JSON

**ntfy-specific:** For `type: ntfy`, the gateway formats the POST body as ntfy expects: `title` from event type, `message` from event data summary, `priority` from event metadata if available.

## Event Delivery to Agents

When an event matches an agent subscription:

1. Gateway wraps the event data into a task for the agent
2. Task is delivered via the existing comms path (POST to comms â†’ WebSocket to body runtime)
3. Body runtime creates a mission-context task from the event
4. Agent processes the event with its mission instructions in the system prompt

The event `data` is included in the task content so the agent knows what triggered it:

```
[Mission trigger: connector jira-ops / issue_created]

New event from jira-ops:
  Key: INC-1234
  Summary: Production database latency spike
  Priority: P1

Process this according to your mission instructions.
```

## Brief Deprecation

The event framework replaces briefs as the task delivery mechanism:

- **Operator â†’ agent task delivery** becomes a DM or channel message routed through the event bus
- **Connector â†’ agent task delivery** becomes a connector event routed through subscriptions
- **Programmatic task delivery** uses the existing comms API (POST message to channel/DM)

The `agency brief` command, `POST /api/v1/agents/\{name\}/brief` endpoint, and `agency_brief` MCP tool are removed. See the brief deprecation spec for full details.

The missions spec's trigger evaluation model (body-side `_is_mission_trigger`) is superseded by the event framework's gateway-side subscription evaluation. With the event framework, the gateway filters events before they reach the body runtime. The body runtime no longer evaluates triggers â€” it simply processes events delivered to it.

The key behavioral change: turn limits are no longer hard-coded per task type. Instead:
- Mission tasks have no artificial turn limit â€” they run until complete or budget-exhausted
- Budget enforcement (LLM call limits, token limits) replaces turn limits as the cost control mechanism
- The budget model is defined in the brief deprecation spec (separate from this spec)

- `event_received` â€” id, source_type, source_name, event_type

Events that match zero subscriptions are logged as `event_unmatched` for debugging and auditing. This satisfies ASK tenet 2 â€” every event is traced even if no subscriber acts on it.

## ASK Tenet Compliance

| Tenet | Compliance |
|-------|-----------|
| **2. Every action traced** | Every event gets a unique ID, logged on receipt and on each delivery. Failed deliveries logged. |
| **3. Mediation complete** | All events flow through the gateway (mediation layer). No direct source-to-agent path. Connector events go through intake â†’ gateway. Channel events go through comms â†’ gateway. |
| **4. Least privilege** | Agents only receive events matching their active subscriptions. Paused missions = deactivated subscriptions = no event delivery. |
| **17. Instructions from verified principals** | Events are data, not instructions. The event tells the agent "something happened." The agent's mission instructions (from the operator) define how to respond. The bus routes data, never instructions. |
| **18. Zero trust default** | Unregistered webhook names rejected (404). Unknown event types match no subscriptions. Webhook authentication via secret token. |
| **19. External can't instruct** | External webhooks produce events (data). The agent's response behavior comes from its mission instructions, not the event content. |

## Scope Boundaries

This spec covers the event framework only. It depends on:
- **Missions spec** â€” mission triggers create subscriptions
- **Meeseeks spec** â€” Meeseeks lifecycle events are platform source events

It does not cover:
- **Budget model** â€” replacing turn limits with LLM call / token / cost budgets (separate spec: brief deprecation)
- **Event source plugins** â€” a plugin API for third-party event sources (future work; v1 sources are built-in only)
- **Event replay** â€” replaying historical events for testing or recovery (future work)
- **Event filtering/transformation** â€” modifying event data before delivery (future work; v1 delivers raw events)

## What This Enables

- Unified activation model â€” every agent wake-up goes through the same path
- Zero LLM calls for idle agents â€” no polling, no keyword scanning, just subscription matching
- External integration via webhooks â€” any system that can POST JSON can trigger agent work
- Scheduled tasks without polling â€” cron expressions fire events, agents respond
- Platform-aware missions â€” agents can react to system state changes
- Operator notifications â€” alerts forwarded to ntfy, Slack, PagerDuty, anything with an HTTP endpoint
- Brief deprecation â€” task delivery becomes message delivery through the event bus
